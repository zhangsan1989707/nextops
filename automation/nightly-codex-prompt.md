运行原则：
- 不要向用户提问，不要等待人工批准；遇到需要决策的地方，按你作为资深工程师的判断选择最保守、最利于项目长期维护的方案。
- 一次只做一个清晰的功能点或架构改造点。
- 每完成一个功能点，必须运行验证：`npm run lint`、`npm run build`、`npm run docker:deploy`、`npm run smoke`。
- 验证通过后，提交一个独立 Git commit，commit message 使用英文祈使句。
- 默认不要 push，除非环境变量或脚本明确要求。
- 不要提交 `.env`、密钥、token、个人凭据或运行日志。
- 如果遇到外部依赖、账号、网络、真实服务器凭据等阻塞，不要停下来询问用户；记录在日志/最终说明中，并切换到下一个可离线完成的任务。
- 如果当前工作区有未提交改动，先判断是否是你本轮生成的；不要回滚用户改动。必要时把你的工作限制在不冲突的文件。
- 每次改动保持小步、可回滚、可验证。

每次开始前先做系统体检：
1. 检查 `git status --short`，不要覆盖用户未提交改动。
2. 用 `rg` 快速扫描当前仍然存在的内存数据源、mock、TODO、placeholder、未使用服务和重复逻辑。
3. 检查 `npm run lint`、`npm run build`、`npm run smoke` 是否能通过；如果失败，优先修复失败项。
4. 检查本次最值得处理的问题，并按“数据正确性 > 可持久化 > 可验证 > 用户体验 > 文档”的顺序排序。
5. 只选择一个最有价值、最可闭环的任务执行；不要一次横跨多个大模块。

当前已知系统问题：
- API 里仍有部分内存数据源，重启会丢失，包括审批工单、包管理、文件管理、快捷指令、多租户大盘等。
- 已落库模块缺少统一审计记录，例如模型、成员、团队、角色、权限的变更历史。
- 前端 `refreshData()` 启动时一次性拉取大量接口，页面增多后会拖慢首屏，并且一个接口失败会影响整体数据初始化。
- Redis 已在 Docker Compose 中启动，但业务代码未使用；要么接入任务队列/缓存，要么暂时移除以降低复杂度。
- AI 诊断、脚本执行、部署计划、文件传输、Agent 安装计划仍是同步 mock，没有任务表、执行记录、状态流转和失败重试。
- 密钥当前能避免 API 明文返回，但用户新增模型的密钥只是本地 demo 持久化，生产前需要加密或接入系统密钥管理。
- API 路由仍集中在 `apps/api/src/index.ts`，继续增长会影响维护；应逐步拆分 routes/services/repositories。
- README 和 docs 中仍有部分 “mock/demo” 描述，功能落库后要同步更新，避免文档与系统状态不一致。
